RISC-V Simulator Architecture Note

从 “指令语义” 切回 “乱序时序”

这个 OOO 实现的核心不是把 RISC-V 指令语义重新写一遍，而是用 DynamicInstPtr 串起 ROB、rename、RS、执行单元、CDB、commit、store forwarding 和 cache 等待。学习时要把“算什么结果”和“什么时候对谁可见”分开。

2-wide 当前 fetch / decode / issue / dispatch / commit 宽度都是 2，见 OOOPipelineConfig。
96 ROB ReorderBuffer::MAX_ROB_ENTRIES 为 96，StoreBuffer 容量也对齐到 96。
48 RS Reservation Station 管理 48 个 entry，以及 ALU / FP / BR / LOAD / STORE 单元。
L1 only 当前没有独立 L2、memory bus 或完整 MSHR，主要是 L1 blocking cache + inflight token。

一眼心智模型

Fetch I$ / branch predictor / fetch buffer
Decode decode + allocate ROB + create DynamicInst
Issue rename + insert Reservation Station
Execute dispatch FU / LSU / cache wait / recovery
Writeback CDB wakeup + physical reg + ROB completed
Commit ROB head only, arch state and real store visibility

每周期执行顺序

OutOfOrderCPU::step() 先 tick I$/D$，再按 commit -> writeback -> execute -> issue -> decode -> fetch 的反向顺序推进。这不是随便排的：同周期老结果先提交/唤醒，新指令再进入前端，读代码时要按这个顺序脑内单步。

ROB、RS、执行单元之间传的是同一个 DynamicInstPtr。所以状态、寄存器重命名结果、 memory info、分支预测信息、异常和 timing cycle 都挂在同一个动态指令对象上。

OOO 乱序核心

Decode 只分配 ROB，不做 rename

DecodeStage 从 fetch_buffer 取一条 instruction，调用 decoder， 然后 reorder_buffer->allocate_entry(...) 创建 DynamicInst。 此时指令状态还是 ALLOCATED，源/目的物理寄存器还没真正绑定。

Rename 解决 WAW / WAR，RAW 仍靠 ready bit 和 CDB

Rename table
逻辑寄存器到物理寄存器映射，整数和 FP 分开维护。

Reservation Station

RS 不是复制一份 instruction，而是持有 DynamicInstPtr。它负责等待操作数、监听 CDB、 按 ready + 执行单元可用 + instruction_id 优先级选择指令。

Reorder Buffer

ROB 保证顺序提交、精确异常，也在 LSU 中扮演 “older store hazard oracle”。 load 完成前会问 ROB：前面有没有 AMO、地址未知 store、或重叠 store。

控制流早恢复

BRANCH / JALR 在 execute 阶段发现预测 next PC 不等于实际 next PC 时，可用 rename checkpoint 清掉 younger ROB/RS/store buffer/CDB/execution units，并把 fetch PC 改到正确路径。

全流水线恢复

trap、FENCE.I、memory-order violation 等路径会更重：清 ROB、RS、rename、store buffer、CDB、执行单元、 cache in-flight，并重启取指。

LSU：这里没有一个传统 LoadStoreQueue 类

这个实现更像 “分布式 LSU”：ROB 做顺序与 older-store hazard 查询，StoreBuffer 做 forwarding 可见性， ExecuteStage 做 replay / cache wait / memory-order violation recovery，CommitStage 做真实 store 可见性。

Load 完成前的决策树

地址不是 Issue 算，而是 Execute semantics 算

Load 用 rs1 + imm 算虚拟地址，再走 translateLoadAddress() 得到物理地址， 写入 unit.load_address 和 MemoryInfo。Store 同理，但还记录 store value。

src/cpu/ooo/execute_semantics.cpp:192, 313

ROB 先回答：前面有没有更老 store 风险

load 完成前调用 get_earlier_store_hazard_info()。返回值不是简单 true/false， 而是 Amo、AddressUnknown、Overlap 或 None。 地址未知 store 有预测器允许 load 推测越过，错了再恢复。

include/cpu/ooo/reorder_buffer.h:27, 157

StoreBuffer 做 byte-mask 级 forwarding

analyze_load_forwarding() 只看比当前 load 更老的 store，按 instruction id 从新到旧合并字节。 完全覆盖可直接返回；部分覆盖会读内存再 merge；有重叠但不能安全合并则 replay。

src/cpu/ooo/store_buffer.cpp:95, 139, 179

D-cache miss 不一定长期占着 load/store unit

如果 cache 访问已经发出且还需要等，执行单元状态会搬到 memory_access_inflight， 同时释放 RS entry 和 dispatch unit。这样另一个 ready load/store 还有机会继续走。

src/cpu/ooo/stages/execute_stage.cpp:619, 545

真实 store 写入只在 Commit 阶段发生

Store execute 阶段可以发布到 StoreBuffer，也可以做 D$ 时序访问，但这不等于体系结构内存已改变。 到 ROB head 且 completed 后，Commit 才调用 l1d_cache->commitStore() 或直接写 Memory。

src/cpu/ooo/stages/commit_stage.cpp:227, 238, 381

不要把 StoreBuffer 当提交队列。 它更像 “forwarding 可见性结构”。Store 对 younger load 可见，不等于已经对体系结构内存可见。
乱序 load 不是无条件越过 store。 它会被 AMO、地址未知 store、重叠 store、MMIO host-comm 串行化等情况挡住或 replay。

Cache / Memory：轻量 L1 blocking 模型

当前已经有什么

  • L1I / L1D 都是 BlockingCache。
  • 默认 32KB、64B line、4-way、hit 1 cycle、miss penalty 20。
  • L1D 默认 max_outstanding_misses = 2，并默认启用 next-line prefetch。
  • cache 直接访问共享 Memory，没有独立 memory bus。

当前还没有什么

  • 没有独立 L2 类或 L2 pipeline。
  • 没有真实 cache-coherent bus / fabric。
  • 没有完整 MSHR：没有同地址合并、target list、refill response 对象。
  • miss 功能上立即装行，时序上靠 pending cycle token 模拟等待。

BlockingCache 的访问模型

A
access / load / fetchInstruction 入口

access() 只建模驻留和时序；load() 还会从 cache line 拼小端值；fetchInstruction() 处理 16/32 位指令。

B
bypass 检查

tohost/fromhost 这种 MMIO 地址直接 bypass cache，并且 execute 侧要求 host-comm 访问等到 ROB head。

C
outstanding miss 限制

如果 outstanding_miss_cycles_.size() >= max_outstanding_misses，返回 blocked=true。

D
hit / miss

hit 更新 LRU；miss 立即 installLine() 从 Memory 填行，返回 hit_latency + miss_penalty。

E
tick 倒计时

CPU 每周期 tick I$/D$，pending cycle 归零后从 token 列表移除。这个 token 就是当前的轻量 pending request。

src/cpu/ooo/cache/blocking_cache.cpp:51, 217, 230, 277, 178

替换与写回

cache 是 set-associative LRU：先找 invalid line，否则选 lru_stamp 最小的 victim。 dirty victim 在 installLine() 里逐字节写回 Memory。Store commit 则用非时序 commitStore() 写 cache line 并同步主存镜像。

blocking_cache.cpp:340, 361, 136

Prefetch

next-line prefetch 由 demand miss 触发，直接安装下一条 line，并统计 already resident、set throttle、 useful hit、unused eviction。CLI 支持开关 --l1d-next-line-prefetch=on|off。

blocking_cache.cpp:382; src/main.cpp:93, 261

两条典型路径走读

1
Decode

分配 ROB entry，创建 DynamicInst。

2
Issue

rename 源寄存器，分配目的物理寄存器，插入 RS。

3
Execute semantics

计算地址、翻译地址、写入 MemoryInfo。

4
Hazard + forwarding

先问 ROB 更老 store 风险，再问 StoreBuffer 能否 full/partial forwarding。

5
L1D

若需要访问 cache，miss 会产生 latency；请求可被搬到 memory_access_inflight。

6
CDB / Writeback / Commit

完成后 CDB 广播，ROB 标 completed，到 head 后提交架构寄存器。

1
Issue

store 本身没有普通整数目的寄存器，但依赖地址源和值源。

2
Execute semantics

地址和值 ready 后写入 MemoryInfo，并发布到 StoreBuffer。

3
D$ 时序访问

store execute 可以用 BlockingCache::access() 建模 miss penalty，但不做真实架构写。

4
Memory order check

如果发现 younger load 曾错误越过地址未知 store，触发 memory-order violation recovery。

5
Commit

到 ROB head 后，commitStore() 写 L1D / Memory，并 retire StoreBuffer entry。

1
Fetch

branch predictor 生成 predicted next PC 和 meta，必要时保存 RAS checkpoint。

2
Issue

BRANCH / JALR 保存 rename checkpoint。

3
Execute resolve

比较 predicted next PC 和 actual next PC。

4
Early recovery

若错预测，flush younger ROB/RS/store buffer/CDB/execution units，恢复 rename checkpoint，PC 指向正确路径。

5
Commit accounting

提交阶段更新 predictor profile、DiffTest 和 retired count；已早恢复的指令不重复清一次。

源码地图

  • include/cpu/ooo/ooo_cpu.h OOO CPU 外壳、阶段对象、对外 CPU 接口。
  • src/cpu/ooo/ooo_cpu.cpp 组件初始化、每周期 step 顺序、L1 cache 默认配置、外部写 invalidate。
  • include/cpu/ooo/cpu_state.h 共享状态：ROB、RS、rename、StoreBuffer、L1I/L1D、执行单元、CDB、perf profile。
  • include/cpu/ooo/dynamic_inst.h 动态指令统一载体；读 OOO 时最值得优先建立对象模型。
  • src/cpu/ooo/stages/decode_stage.cpp fetch buffer 到 ROB allocation；注意这里还没有 rename。
  • src/cpu/ooo/stages/issue_stage.cpp rename、串行化约束、插入 RS、分支 rename checkpoint。
  • src/cpu/ooo/stages/execute_stage.cpp 执行单元推进、RS dispatch、load/store hazard、cache wait、inflight、recovery。
  • src/cpu/ooo/stages/writeback_stage.cpp CDB 消费，唤醒 RS、更新物理寄存器、标记 ROB completed。
  • src/cpu/ooo/stages/commit_stage.cpp ROB head 顺序提交、架构寄存器、store/AMO 可见性、trap/flush/DiffTest。
  • src/cpu/ooo/execute_semantics.cpp 指令语义入口；load/store 地址计算与地址翻译也在这里。
  • include/cpu/ooo/store_buffer.h StoreBuffer forwarding API 和 full/partial/blocked 分类。
  • src/cpu/ooo/cache/blocking_cache.cpp L1 cache hit/miss、LRU、dirty writeback、prefetch、pending cycles。
