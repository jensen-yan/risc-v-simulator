# ExecPlan 指南

这份文档定义本仓库中复杂任务的执行计划（ExecPlan）应该如何编写、维护和使用。

它适用于这类工作：

- 无法在一两轮对话内稳定完成的任务
- 需要跨多个步骤、文件、实验或验证回合的工作
- 需要保留背景、证据、阶段结论和下一步动作的分析任务
- 容易丢失上下文的长周期调试、性能分析、重构和分阶段实现

如果任务只是很小的代码改动、单点修复、简单文案调整，通常不需要单独写 ExecPlan。

## 1. ExecPlan 是什么

ExecPlan 不是随手记的 TODO，也不是只有几条 checklist 的轻量备注。

它应该能回答这些问题：

- 当前要解决的问题是什么
- 为什么值得做
- 已经确认了什么
- 还不确定什么
- 下一步具体做什么
- 如何判断这轮工作达成
- 中途有哪些重要发现或方向调整

理想状态是：即使接手的人不了解之前的聊天记录，只读当前 ExecPlan 也能继续推进任务。

## 2. 什么时候应该使用

满足以下任一条件时，优先新建或更新 ExecPlan：

1. 任务预计跨多轮对话或多次实验
2. 同时包含研究、调试、实验、实现中的两项及以上
3. 涉及多个模块、脚本、统计口径或验证入口
4. 当前存在明显不确定性，需要先验证假设
5. 需要沉淀“为什么这么做”的决策记录，而不只是“改了什么”
6. 任务可能暂停后再继续

典型例子：

- SPEC06 / checkpoint / score / topdown 分析
- OOO 流水线、flush、redirect、仲裁或可见性问题调试
- 长链路性能瓶颈定位
- 跨模块重构
- 需要分阶段推进的新能力

## 3. 编写原则

### 3.1 自包含

不要假设读者记得上下文，不要写“和前面讨论的一样”。

如果某段背景会影响后续判断，就直接写进当前文档。

### 3.2 结果导向

不要只写“改某个函数”或“补某个字段”。

还要说明：

- 预期解决什么问题
- 外部能观察到什么变化
- 用什么证据判断结果成立

### 3.3 证据导向

对分析类任务，必须明确区分：

- 已确认事实
- 当前假设
- 未解决问题
- 支撑结论的证据

不要把猜测写成事实。

### 3.4 持续更新

ExecPlan 是活文档，不是一次性写完就不再维护的备忘录。

推进过程中要持续更新：

- 当前状态
- 新发现
- 决策变化
- 下一步动作

### 3.5 先写为什么

实现细节可以逐步展开，但关键决策必须说明理由。

后续接手的人需要知道为什么选当前路径，而不是别的路径。

## 4. 推荐位置

对复杂任务，优先一项任务一份 ExecPlan，而不是把所有内容塞进同一个大文档。

本仓库默认把 ExecPlan 当作本地工作文档处理，推荐目录结构：

```text
docs/
  exec-plans/
    active/
    completed/
    blocked/
```

含义：

- `active/`: 当前推进中的任务
- `completed/`: 已完成但希望本地保留的任务
- `blocked/`: 因外部条件阻塞的任务

命名建议简洁明确，例如：

- `docs/exec-plans/active/spec06-score-bottleneck-analysis.md`
- `docs/exec-plans/active/ooo-store-ordering-debug.md`
- `docs/exec-plans/active/memory-observability-gap.md`

默认情况下，ExecPlan 实例不要求提交到 git。
只有在明确需要共享、评审、交接或保留正式决策记录时，才将某份 ExecPlan 纳入版本管理。

## 5. 推荐结构

每份 ExecPlan 通常至少包含以下部分。

## 标题

标题优先用“动作 + 对象”，避免模糊命名。

例如：

- 分析 SPEC06 分数与 topdown 主瓶颈
- 定位 OOO 流水线的前端供给问题
- 补齐 memory-first 路线的可观测性

## 背景与目标

用几段话说明：

- 当前问题是什么
- 为什么重要
- 这轮希望得到什么结果
- 如何观察到结果

先讲目标价值，再讲实现细节。

## 当前已知信息

记录已经确认的事实和约束，例如：

- 相关文件、模块、脚本、路径
- 当前行为与预期差异
- 已看到的日志、counter、trace、实验结果
- 环境和输入约束

这一节不写未经验证的猜测。

## 假设与开放问题

把仍不确定的内容显式列出来，例如：

- 当前怀疑瓶颈主要在前端供给，而不是分支预测
- 还未确认 `topdown-lite` 百分比是否足够支持结构级结论
- 需要验证某组 benchmark 是否共享同一 stall 模式

## 计划步骤

按顺序列出下一步动作。每一步尽量写成“动作 + 目的 + 预期产出”。

例如：

1. 对齐分数口径，确认比较对象是否同 coverage
2. 汇总 benchmark 级 stall / IPC / topdown 画像
3. 下钻最低分 benchmark 的点级分布
4. 判断主瓶颈属于前端、后端、内存还是错误推测
5. 形成下一轮优化建议和验证入口

## 验证标准

必须说明怎样算完成。

验证可能包括：

- 测试通过
- 统计结果与预期一致
- 复现到根因
- 候选假设被排除
- 得到可执行的下一轮优化建议

对分析任务而言，“完成”也可以是：

- 根因被明确缩小到某类机制
- 若干错误假设被证伪
- 输出了清晰的下一步决策

## 进展

进展要持续更新，建议使用带时间的 checkbox：

- [x] 2026-04-15 14:20 对齐 `0.3c` coverage 口径
- [x] 2026-04-15 15:10 汇总 benchmark 级 score / stall 画像
- [ ] 下钻最低分 benchmark 的点级 IPC 和 stall 分布
- [ ] 形成下一轮优化建议

如果只是部分完成，要写清楚已经完成什么、还差什么。

## 新发现与意外点

记录推进过程中出现的重要新信息，例如：

- 之前的理解是错的
- 文档和代码口径不一致
- 某个 counter 不能直接用于结构性判断
- 某组 benchmark 的画像和预想不同

## 决策记录

当方向发生变化或做出关键取舍时，要写清楚：

- 决策内容
- 做出时间
- 依据是什么
- 放弃了哪些备选方案

## 6. 与仓库其他文档的关系

- `README.md`：构建、运行、测试入口
- `ARCHITECTURE.md`：模块边界、主执行路径
- `tasks/`：任务背景、阶段记录、较稳定的专项报告
- `PLANS.md`：说明 ExecPlan 应该怎么写
- `docs/exec-plans/`：当前具体任务的执行记录

避免在多个地方复制同一份长背景，减少漂移。

## 7. 本仓库中的建议做法

- 对复杂分析任务，先建 ExecPlan，再继续实验或编码
- 每次得到关键结果时，优先回写 ExecPlan，再展开下一步
- 如果当前任务已经有 `tasks/` 文档，ExecPlan 只记录本轮推进所需的动态上下文，不重复整篇专项报告
- 如果某份 ExecPlan 需要正式长期保留，再转化为 `tasks/` 或其他受版本管理的文档
